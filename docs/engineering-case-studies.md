# Архитектурные кейсы и рефакторинг технического долга (.NET / EF Core)

В данном документе собраны реальные инженерные кейсы из моей практики — опыт дебага сложных архитектурных проблем.

---

## Кейс: эволюция контекста текущего пользователя (DI-гонка → утечка абстракции → SOLID-разделение)

Три последовательных этапа работы над одной и той же областью — определением "кто выполняет действие" для аудита и авторизации. Каждый этап устранял конкретный баг, но заодно обнажал следующую проблему в самой конструкции решения; итоговая архитектура — в конце документа.

### Этап 1. Устранение гонки условий в DI (плавающий баг с подменой контекста пользователя)

#### 📌 Situation (Ситуация)
В веб-приложении была реализована логика автоматического аудита (проставление `CreatorId` при записи в БД). Для этого использовался интерфейс `ICurrentUser`, который имел две реализации: `CurrentUser` (веб-пользователь, читающий данные из HTTP-контекста) и `SystemUser` (системная учетная запись для фоновых задач).

Выбор реализации происходил на этапе регистрации в DI-контейнере через фабричный метод:
```csharp
services.AddScoped<ICurrentUser>(sp =>
{
    var accessor = sp.GetRequiredService<IHttpContextAccessor>();
    return accessor.HttpContext != null
        ? sp.GetRequiredService<CurrentUser>()
        : sp.GetRequiredService<SystemUser>();
});
```

#### 🎯 Task (Задача)
Обеспечить стабильное и изолированное автоматическое определение автора изменений как для веб-запросов, так и для фоновых воркеров.

#### 🛠️ Action (Действие и Расследование)
На этапе тестирования был обнаружен критический плавающий баг: авторизованные веб-пользователи периодически определялись системой как `SystemUser`.

**Причина бага:** Нарушение жизненного цикла (`Lifecycle Timing`) IoC-контейнера Microsoft.Extensions.DependencyInjection и конвейера Middleware.
Если фабрика `AddScoped` вызывалась в конвейере запроса *до* завершения инициализации `HttpContext` (например, в раннем кастомном Middleware логирования), она выполняла ветку `else`, фиксировала `SystemUser` и **кэшировала этот экземпляр на весь Scope текущего HTTP-запроса**. Последующие вызовы в контроллерах и триггерах СУБД получали системные права вместо прав пользователя.

**Решение:** Логика выбора стратегии перенесена из момента инициализации DI в рантайм-геттеры с помощью паттерна **Динамический Прокси** — выбор откладывается до фактического обращения к свойству, а не фиксируется один раз при создании scope.

```csharp
public class RuntimeCurrentUser : ICurrentUser
{
    private readonly IHttpContextAccessor _accessor;
    private readonly IServiceProvider _serviceProvider;

    public RuntimeCurrentUser(IHttpContextAccessor accessor, IServiceProvider serviceProvider)
    {
        _accessor = accessor;
        _serviceProvider = serviceProvider;
    }

    private ICurrentUser CurrentStrategy => _accessor.HttpContext != null
        ? _serviceProvider.GetRequiredService<CurrentUser>()
        : _serviceProvider.GetRequiredService<SystemUser>();

    public Guid UserId => CurrentStrategy.UserId;
    public bool IsAuthenticated => CurrentStrategy.IsAuthenticated;
}
```

#### 🚀 Result (Результат)
* Полностью устранен плавающий баг с потерей контекста авторизации.
* Регистрация в DI стала прямолинейной и безопасной (`services.AddScoped<ICurrentUser, RuntimeCurrentUser>()`).
* Но сам факт, что `ICurrentUser` вынужден на каждое обращение решать, веб он или система — сигнал, что интерфейс всё ещё делает две разные вещи (см. Этап 3).

### Этап 2. Предотвращение уязвимости "Silent Failure" (утечка абстракции ClaimsPrincipal)

#### 📌 Situation (Ситуация)
Похожая по духу проблема обнаружилась в соседнем компоненте той же зоны ответственности — определении личности вызывающего. Доступ к контексту пользователя осуществлялся через интерфейс `IUserAccessor`, который возвращал сырой объект .NET `ClaimsPrincipal User { get; }`. Чтение ID пользователя происходило через методы расширения (`_userAccessor.User.GetUserId()`), внутри которых использовался жесткий метод `Guid.Parse()`.

#### 🎯 Task (Задача)
Изолировать бизнес-логику приложения от инфраструктурных типов веб-сервера (ASP.NET), обеспечить безопасность типов и отказоустойчивость при парсинге токенов.

#### 🛠️ Action (Действие)
Я выявил две критические проблемы в текущей реализации:
1. **Утечка абстракции:** Передача `ClaimsPrincipal` в другие слои приводила к зависимости домена от веба. При попытке вызвать этот метод в асинхронном фоновом потоке после завершения HTTP-запроса, контекст уничтожался (`Disposed`), вызывая `ObjectDisposedException`.
2. **Антипаттерн "Silent Failure" / Потенциальная уязвимость:** Была идея при пустом контексте возвращать дефолтный системный `Guid`. Я заблокировал это решение, так как «тихий» возврат системного ID маскирует ошибки конфигурации бэкенда. Если метод защиты ожидает реального пользователя, но вызывается вне HTTP-сессии, система обязана упасть громко (**Fail-Fast**), иначе возникнет дыра в безопасности.

**Решение:** Был выполнен рефакторинг интерфейса в чистую бизнес-абстракцию. Вся работа с клеймами и безопасным парсингом (`Guid.TryParse`) инкапсулирована внутри реализации, а вычисления перенесены в ленивые геттеры с локальным кэшированием (Backing Field) в рамках запроса для высокой производительности.

```csharp
public class HttpUserAccessor : IUserAccessor
{
    private readonly IHttpContextAccessor _httpContextAccessor;
    private Guid? _cachedUserId;

    public HttpUserAccessor(IHttpContextAccessor httpContextAccessor) => _httpContextAccessor = httpContextAccessor;

    public Guid UserId
    {
        get
        {
            if (_cachedUserId.HasValue) return _cachedUserId.Value;

            var idStr = _httpContextAccessor.HttpContext?.User?.FindFirstValue(ClaimTypes.NameIdentifier);
            if (Guid.TryParse(idStr, out var guid))
            {
                _cachedUserId = guid;
                return guid;
            }
            throw new UnauthorizedAccessException("Контекст пользователя отсутствует или не валиден.");
        }
    }
}
```

#### 🚀 Result (Результат)
* Бизнес-логика полностью изолирована от веб-специфики, код стал легко тестируемым (Unit Testing) без необходимости мокать `ClaimsPrincipal`.
* Реализован принцип **Fail-Fast**, защищающий приложение от выполнения критических операций под дефолтными/системными правами в случае сбоя контекста.

### Этап 3. Переосмысление полиморфизма и рефакторинг под SOLID (нарушение LSP)

#### 📌 Situation (Ситуация)
Для решения задач из *Этапа 1* использовался единый интерфейс `ICurrentUser`, объединяющий `CurrentUser` (веб) и `SystemUser` (система).

#### 🎯 Task (Задача)
Провести ревью архитектуры приложения на соответствие принципам SOLID и исправить концептуальные ошибки проектирования.

#### 🛠️ Action (Действие)
В процессе анализа я пришел к выводу, что текущая реализация полиморфизма является неудачной. Объединение веб-пользователя и фонового процесса под одним интерфейсом нарушало **Принцип подстановки Лисков (Liskov Substitution Principle — LSP)**.

Вызывающий код (например, триггер БД) ожидал от интерфейса предсказуемого поведения, но реализации кардинально отличались: одна могла выбросить исключение или вернуть `null` при отсутствии сессии, а вторая гарантированно возвращала зашитый ID. Это вынуждало строить хрупкие фабрики-прокси в DI-контейнере.

**Решение:** Я применил **Принцип разделения интерфейса (Interface Segregation Principle — ISP)**. Единый интерфейс был упразднен. Вместо него созданы два честных контракта, четко разделяющих контексты выполнения:

```csharp
// Контракт для веб-слоя (Действия человека в браузере)
public interface IWebUserContext
{
    Guid UserId { get; } // Выбрасывает Exception, если контекста нет (Fail-Fast)
    bool IsAuthenticated { get; }
}

// Контракт для инфраструктурного слоя (Фоновые задачи, очереди, планировщики)
public interface ISystemContext
{
    Guid SystemComponentId { get; }
    string ComponentName { get; }
}
```

#### 🚀 Result (Результат)
* Код избавился от скрытой «магии» на уровне DI-контейнера и стал прозрачным.
* Полностью соблюдены принципы LSP и ISP. Разработчик на этапе написания кода четко понимает, с каким контекстом (веб или воркер) работает конкретный сервис, что исключает появление логических багов в продакшене.
* Но разделение породило новый вопрос: инфраструктурному коду (в частности, `AuditInterceptor`), которому нужен только `Guid` автора, а не полноценный веб- или системный контракт, теперь неоткуда получить это значение единообразно — см. итог ниже.

## Итог: единый прозрачный контекст личности

После разделения контрактов на `IWebUserContext` и `ISystemContext` (Этап 3) возникла необходимость поставлять `Guid` автора изменений в общий интерсептор базы данных (`AuditInterceptor`), которому не важно, веб это или система — важен только факт "кто это сделал".

Я рассматривал два пути решения: функциональный (регистрация делегата `Func<Guid>` в DI) и контрактный (введение узкого объединяющего интерфейса).

Выбор сделан в пользу контрактного подхода: введён интерфейс `IIdentityContext`, содержащий ровно одно общее для всех процессов свойство — `Guid ActorId { get; }`. Оба существующих контракта проецируются в него через ту же схему динамического прокси, что и в Этапе 1, только теперь она выбирает не между двумя *реализациями одного интерфейса*, а между двумя *разными контрактами*, приводя их к общему знаменателю:

```csharp
public class RuntimeIdentityContext : IIdentityContext
{
    private readonly IHttpContextAccessor _accessor;
    private readonly IServiceProvider _serviceProvider;

    public RuntimeIdentityContext(IHttpContextAccessor accessor, IServiceProvider serviceProvider)
    {
        _accessor = accessor;
        _serviceProvider = serviceProvider;
    }

    public Guid ActorId => _accessor.HttpContext != null
        ? _serviceProvider.GetRequiredService<IWebUserContext>().ActorId
        : _serviceProvider.GetRequiredService<ISystemContext>().ActorId;
}
```

### 🚀 Итоговый результат
* `AuditInterceptor` и прочая инфраструктура зависят только от `IIdentityContext` — минимального контракта, который физически гарантирован у любого инициатора изменений в системе, без Email, ComponentName или прочих специфичных для веба/воркера полей.
* `IWebUserContext` и `ISystemContext` остались honest-контрактами для своих слоёв (ISP из Этапа 3), не потеряв при этом общей точки входа для аудита.
* Гонка условий из Этапа 1 не вернулась: выбор конкретной реализации по-прежнему откладывается до момента обращения (рантайм-геттер), а не фиксируется на старте scope.
